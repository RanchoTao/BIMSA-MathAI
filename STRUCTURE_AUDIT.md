# 仓库结构审计（第一阶段）

## 1) 当前目录结构（审计时点）
当前仓库以“按学期目录 + 课程文件散放”为主，核心形态如下：

- 学期目录：`2025秋/`、`2026冬/`、`2026春/`、`2026秋/`、`2027春/`、`2027秋/`、`2028春/`
- 顶层文件：`README.md`、`LICENSE`、若干 PDF（如 `求真.pdf`、`数学.pdf`、`人工智能.pdf`）
- 各学期内包含课程文件、作业文件、课表图片/表格、压缩包、文档等

文件类型混合明显：`pdf`、`docx`、`pptx`、`xlsx`、`csv`、`zip`、`jpg/png/webp`、`py`、`md` 共同存在。

## 2) 主要问题

### 2.1 信息架构问题
- 以时间（学期）为主轴，但缺少跨学期“按主题聚合”的知识入口。
- 缺少统一 `docs/` 文档层，学习方法、课程知识、科研训练、工具教程未分层。
- 仓库首页 README 偏个人叙述，尚未形成知识库导览。

### 2.2 命名与规范问题
- 中文目录、英文目录与无扩展名文件混用（如多个名为 `1` 的文件）。
- 部分文件名包含空格、日期、个人署名，不利于长期 URL 稳定与检索一致性。
- 课程目录命名规则不统一，如“中文课程名 + 作业序号”与“英文缩写”并存。

### 2.3 内容组织问题
- 同类内容分散：例如机器学习资料同时出现在不同学期目录。
- 作业、讲义、图片、课表等未按统一子目录分拣。
- 存在明显“个人归档型文件”，但尚未与“可复用知识内容”分离。

## 3) 重复 / 混乱 / 命名不统一示例
- 占位文件泛化：多处出现名为 `1` 的文件，语义不明确。
- 同一课程跨学期重复：如数学分析、机器学习在不同学期目录都有资料。
- 文件命名风格混杂：
  - `Disc-Math-1.pdf`（英文 + 序号）
  - `机器学习第10次作业.docx`（中文 + 序号）
  - `数据挖掘作业 陶圣叶 2026年4月7日.md`（含空格+姓名+日期）

## 4) 建议的新信息架构（第一阶段已落地骨架）
建议采用“主题分层 + 课程标准目录 + 模板驱动”的架构：

- `docs/survival-guide/`：新生生存与学习方法
- `docs/courses/<course>/`：课程知识库（统一 `README.md + notes/assignments/exams/resources`）
- `docs/research/`：论文阅读、学术写作、研究流程
- `docs/tools/`：Git/GitHub、LaTeX、VSCode、Python 等工具链
- `docs/lifestyle/`：时间管理、身心健康、认知模型
- `templates/`：课程页、笔记、论文阅读、贡献模板
- `archive/`：历史资料分批迁移缓冲区

## 5) 迁移风险评估

### 5.1 链接与引用风险
- 直接重命名或整体搬迁会导致历史链接失效。
- 建议先建新入口，再小步迁移并记录映射表。

### 5.2 语义丢失风险
- 文件名中包含“作业次数/日期/课程语境”时，粗暴改名会丢失信息。
- 建议先在目标目录新增 `README` 索引，再执行重命名。

### 5.3 合规风险
- 课程作业、讲义、考试相关文件可能涉及传播边界。
- 迁移时应增加脱敏与版权检查步骤。

### 5.4 协作风险
- 在未形成统一模板和命名规则前批量迁移，会加剧后续冲突。
- 当前阶段以“建立规范 + 保留旧资料位置”为优先策略是合理的。

- 未执行：旧资料批量迁移与重命名（按要求暂缓）。
- 建议下一阶段：先做“课程映射清单”，再按课程逐步迁移高价值资料。
